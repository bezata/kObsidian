# Tool surface

kObsidian exposes **62 MCP tools across 15 namespaces** plus 4 resources
and 3 prompts. The canonical, always-up-to-date list is generated by
`bun run inventory` into **[`tool-inventory.json`](tool-inventory.json)**.

> v0.2.5 consolidated the surface from ~90 tools to 62. See
> [`../CHANGELOG.md`](../CHANGELOG.md) and [`MIGRATION.md`](MIGRATION.md)
> for the full rename/merge table.

## Annotation summary

Every tool declares the full 4-hint MCP 2025-11-25 annotation set
(`readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint`),
so MCP clients (Claude Desktop, Claude Code, Cursor 2026+, etc.) can decide
auto-approval behavior without falling back to defaults.

| Annotation constant | Shape | Typical use |
|---|---|---|
| `READ_ONLY` | read-only, idempotent, closed-world | `notes.read`, `tags.list`, `blocks.read`, all `links.*`, `wiki.query` |
| `READ_ONLY_OPEN_WORLD` | read-only + open-world | `dataview.query` family (live plugin) |
| `ADDITIVE` | writes new content, not idempotent | `notes.create`, `notes.edit`, `tags.modify(add)`, `kanban.card(add)` |
| `ADDITIVE_OPEN_WORLD` | UI-additive via REST | `workspace.openFile`, `workspace.navigate`, `workspace.toggleEditMode` |
| `IDEMPOTENT_ADDITIVE` | additive but idempotent | `notes.frontmatter`, `tags.modify`, `dataview.fields.write`, `wiki.init` |
| `IDEMPOTENT_DESTRUCTIVE` | overwrites in place, idempotent | `notes.edit(replace)`, `blocks.update`, `marp.update`, `wiki.indexRebuild`, `notes.move` |
| `DESTRUCTIVE` | destructive, not idempotent | `notes.delete`, `commands.execute` |

## Resources

URI-addressable data any MCP client can browse without calling tools:

```
kobsidian://wiki/index                       wiki/index.md (catalog)
kobsidian://wiki/log                         wiki/log.md (chronological)
kobsidian://wiki/schema                      wiki/wiki-schema.md (contract)
kobsidian://wiki/page/{+path}                any Sources/Concepts/Entities page
```

The `wiki/page/{+path}` template has a `list` callback that enumerates
every page, so a client can populate a file-tree UI without tool calls.

## Prompts

Server-side prompt templates for clients that don't consume the
`skills/` files:

- `ingest-source` — wraps `wiki.ingest` + the `proposedEdits` loop
- `answer-from-wiki` — wraps `wiki.query` + cited synthesis
- `health-check-wiki` — wraps `wiki.lint` + triaged fix proposals

Each takes a Zod-validated `argsSchema` that the client renders as a form.

## Structured output

Every tool declares an `inputSchema` and `outputSchema` built from named
Zod schemas in [`../src/schema/`](../src/schema/). Root argument objects
use `.strict()` (emits JSON Schema `additionalProperties: false`), every
field carries `.describe()`, and complex tools carry `inputExamples` that
are rendered into the registered tool description — so Anthropic clients,
Glama, and other LLMs see inline worked examples without extra round trips.

Clients that understand `structuredContent` (MCP 2025-06-18 onward) get
typed JSON directly; older clients fall back to the stringified `content`
block.

## Regenerating the inventory

```bash
bun run inventory   # writes docs/tool-inventory.json
```

The inventory includes each tool's `name`, `title`, `description`, and
`annotations`. It's committed so docs and npm consumers see the full
surface without cloning.
