# Tool surface

kObsidian exposes **90 MCP tools** across 12 namespaces plus 4 resources
and 3 prompts. The canonical, always-up-to-date list is generated by
`bun run inventory` into **[`tool-inventory.json`](tool-inventory.json)**.

| Namespace | Count | What it does |

| **Total** | **90** | |

## Annotation summary

Every MCP client that reads tool annotations (Claude Desktop, Claude Code,
Cursor 2026+, etc.) uses these to decide auto-approval behavior.

| Annotation | Count | Effect |
|---|---|---|
| `readOnlyHint: true` | 47 | Clients can auto-approve; no vault mutation |
| `destructiveHint: true` | 6 | Clients prompt more firmly; examples: `notes.delete`, `notes.move`, `notes.moveFolder`, `canvas.removeNode`, `commands.execute`, `wiki.indexRebuild` |
| `idempotentHint: true` | 12 | Safe to retry; same call twice = same result |
| `openWorldHint: true` | 16 | Reaches outside the vault (Obsidian REST API) |

The rest are unannotated — clients fall back to default prompting (treat
as neither read-only nor destructive).

## Resources

URI-addressable data any MCP client can browse without calling tools:

```
kobsidian://wiki/index                       wiki/index.md (catalog)
kobsidian://wiki/log                         wiki/log.md (chronological)
kobsidian://wiki/schema                      wiki/wiki-schema.md (contract)
kobsidian://wiki/page/{+path}                any Sources/Concepts/Entities page
```

The `wiki/page/{+path}` template has a `list` callback that enumerates
every page, so a client can populate a file-tree UI without tool calls.

## Prompts

Server-side prompt templates for clients that don't consume the
`skills/` files:

- `ingest-source` — wraps `wiki.ingest` + the `proposedEdits` loop
- `answer-from-wiki` — wraps `wiki.query` + cited synthesis
- `health-check-wiki` — wraps `wiki.lint` + triaged fix proposals

Each takes a Zod-validated `argsSchema` that the client renders as a form.

## Structured output

Every tool that returns data declares an `outputSchema` and returns
`structuredContent` alongside the conventional `content` array. Clients
that understand structured output (2025-06-18 spec onward) get typed JSON
directly; older clients fall back to the stringified `content` block.

## Regenerating the inventory

```bash
bun run inventory   # writes docs/tool-inventory.json
```

The inventory includes each tool's `name`, `title`, `description`, and
`annotations`. It's committed so docs and npm consumers see the full
surface without cloning.
